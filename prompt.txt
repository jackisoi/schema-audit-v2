IMPORTANT — BLOCKED PAGE DETECTION:
If the page content explicitly contains the exact text "Enable JavaScript and cookies to continue", or "cf-browser-verification", or "Just a moment" together with "Cloudflare":
- In Executive Summary: open with "⚠️ PAGE BLOCKED — This page is protected by Cloudflare or a JavaScript gate. Content and schemas could not be extracted."
- Do NOT write "no schemas found" — write "schemas could not be verified due to page block"
- Do NOT list missing schemas as critical issues — they may exist but are hidden behind the block
- Still provide schema recommendations based on the URL, page type, and level alone

STEP 1 - Detect schema manager:
Examine the existing schemas and identify what is generating them:
- If schemas appear inside an @graph array → managed by an SEO plugin or platform
- Common platforms: Yoast SEO, RankMath, AIOSEO (WordPress), Shopify, Wix, Squarespace, Webflow
- If schemas are standalone <script> blocks without @graph → Custom / Manual code
- State clearly at the start of section 3: "Schema manager detected: [name]"

Classify each existing schema type as:
- MANAGED: auto-generated by the detected platform → configuration instructions only, NO code block
- CUSTOM: manually coded → provide full replacement code block if changes needed

STEP 2 - Analyze existing schemas:
For each schema found, identify:
- Wrong type for this page (e.g. LocalBusiness instead of Hotel)
- Missing required properties
- Duplicate schemas
- Broken or inconsistent @id values
- SearchAction present but no search form detected on page → flag for removal
If no existing schemas: state "No existing schemas found."

STEP 3 - Cross-page awareness:
- Level 1 (home page): implement WebSite, Organization, WebPage
- Any other level: do NOT implement Organization or WebSite
- Sub-page status for this URL: __SUBPAGE_NOTE__
- If sub-page: reference parent entity via its @id — do NOT redefine it
- If RECOMMENDED SCHEMAS FROM PARENT PAGES are provided above:
  use those @ids EXACTLY and do not contradict any decision made there

STEP 4 - Produce the report as a JSON array of Notion blocks:

1. Executive Summary (heading_2 + paragraph)
   Schema health score: Poor / Fair / Good / Excellent
   What was found, what is missing, urgent issues with ⚠️
   Do NOT mention accessibility, alt text, WCAG, or screen readers
   Do NOT include any Minor Observations in this section

2. Schemas to Implement (heading_2)
   One complete code block per schema — CUSTOM schemas only (MANAGED excluded)
   Label each block:
   - NEW — schema that does not exist on the page yet
   - REPLACES [existing type] — full replacement for an existing custom schema
   If a schema replaces an existing one: provide the complete final version, not a diff
   This section combines both new schemas AND replacements — do NOT split into two sections

3. [Platform name] Configuration (heading_2)
   Text instructions only — NO code blocks here
   For each MANAGED schema: what to configure in the platform settings
    If a managed schema has issues that the plugin cannot resolve natively:
    - FIRST: instruct the client to check if the plugin can be configured to fix the issue
      (e.g. adjust settings, re-enter content as plain text, enable/disable a specific option).
    - ONLY IF the plugin cannot resolve it: instruct to disable and use the custom implementation.
      write: "First, check whether your SEO plugin can be configured to resolve this issue natively.
      If not, disable this auto-generated schema in plugin settings — see custom implementation above."
    This rule applies to ALL schema types — FAQPage, Product, LocalBusiness, BreadcrumbList, and any other.
   If SearchAction exists in the managed WebSite schema but no search form was detected:
     instruct to disable SearchAction in plugin settings.
   If no managed schemas exist on this page: omit this section entirely

   After all text instructions, add:
   heading_3: "Complete Implementation Code"
   paragraph: "This block replaces ALL required schemas for this page. Paste it into your site's <head> section via a custom code injection plugin. Before publishing: fill in all // REQUIRED fields and adjust any dynamic variables (e.g. page-specific URLs, names, dates) to match the actual content shown to visitors. You are responsible for verifying all values before going live."
   ONE single code block (language: html) combining ALL schemas from section 2.
   CRITICAL: This must be exactly ONE code block — do NOT split it, even if it exceeds 1900 characters.
   The 1900-character limit does NOT apply to this combined implementation block.
   Format:
   <script type="application/ld+json">
   [
     { ...schema 1... },
     { ...schema 2... }
   ]
   </script>
  After the combined code block, always add this warning paragraph:
"⚠️ Static code notice: This code is not dynamic. Any changes to the page content
(questions, answers, products, business hours, etc.) must be manually updated in this
code as well. Failing to maintain it will cause a mismatch between what visitors see
and what Google reads." 

4. Action Items (heading_2)
   A numbered list of concrete tasks for the client to implement, in priority order.
   Each item must be a specific action — not a summary of what was already written.
   Examples: "Disable FAQPage in Yoast settings", "Add telephone to Organization in plugin",
   "Paste FAQPage code via WPCode plugin"
   Do NOT repeat explanations from previous sections — tasks only.
   Do NOT include accessibility-related content.
   If all required actions are already covered in the sections above with no additional
   implementation steps needed: omit this section entirely.

5. Minor Observations (heading_2) — OPTIONAL
   Include ONLY if there are non-critical cosmetic or low-priority issues.
   Examples: 15 separate SiteNavigationElement objects instead of ItemList,
   minor @id formatting inconsistencies that do not affect functionality.
   Rules:
   - If you are unsure whether an issue is critical or minor → do NOT include it here.
     Omit it entirely from this report. It will surface in the QA report.
   - Do NOT include anything from this section in the Executive Summary.
   - If no minor issues: omit this section entirely.

Rules:
- Return a raw JSON array only, no surrounding text
- All report text in English
- Schema JSON-LD field values must match the actual page language
- @id naming conventions — always use exactly:
  #org (Organization), #website (WebSite), #webpage (WebPage), #breadcrumb (BreadcrumbList)
  For page-specific entities: /path/#type (e.g. /nucha/#hotel, /nucha/#faq)
- CRITICAL: If RECOMMENDED SCHEMAS FROM PARENT PAGES are provided, use those @id values
  exactly. Never invent alternative @id values for entities defined in parent pages.
- Never invent or estimate property values — use: // REQUIRED: [description]
- Integer-type properties (numberOfItems, position, ratingValue, etc.) must be OMITTED entirely if the value is unknown — do NOT use a string placeholder for integer fields
- Never write field values not explicitly present on the page
- Never add starRating or aggregateRating unless confirmed on the page
- telephone property must always use E.164 format: + followed by country code and digits only, no hyphens, no spaces, no parentheses (e.g. +972544556217, not +972-54-4556217)
- Never include sameAs: [] — if social URLs unknown, omit sameAs entirely
- Never recommend SearchAction unless a search input is confirmed on the page
- If SearchAction exists in existing schemas but no search form detected: flag as issue to disable
- Do NOT add FAQPage unless real Q&A content exists on the page
- Do NOT add Organization or WebSite on non-homepage pages
- If FAQPage exists: review all Q/A pairs for swapped content (answer in name field, question in
  text field). List any swapped pairs by number. Tell the client the exact count of questions.
- If FAQPage answer text contains HTML tags (<p>, <a>, etc.): flag ONLY as an Action Item — do NOT mention this in the Executive Summary, do NOT mark it as critical, do NOT use ⚠️ for this issue. Use this exact explanation:
  "The acceptedAnswer.text fields contain raw HTML tags. Schema.org defines this field as plain Text, and Google's Rich Results Test may show a warning for HTML content. While Google does not strictly forbid HTML in this field, plain text is the accepted best practice — HTML tags may appear as literal characters in search results or cause partial content to be filtered out. Strip all <p>, <a href>, and other HTML markup from the answer text and replace with plain text only."
- For FAQPage with more than 3 questions: write a summary note, no full schema code
- Each individual code block must not exceed 1900 characters — split if needed
- Do NOT include any content about: alt text, WCAG, screen readers, color contrast,
  aria labels, keyboard navigation, or any other accessibility topic
- Any issue classified as Minor Observation must appear ONLY in the Minor Observations
  section. It must NOT appear in the Executive Summary, Schemas to Implement, or
  Action Items. Never mention the same issue in more than one section.
- SiteNavigationElement consolidation (multiple separate objects vs. single ItemList)
is ALWAYS a Minor Observation only — never a Critical Issue, never in Executive Summary,
never in Schemas to Implement, and no implementation code.
ABSOLUTE: Do NOT mention SiteNavigationElement in the Executive Summary under any circumstances — not even with a ⚠️, not even as a passing note. It belongs ONLY in Minor Observations.
Write only:
"The site outputs [N] separate SiteNavigationElement objects. Consolidating them into
a single ItemList with ListItem children would be more efficient, but this does not
affect functionality."
